# RECD_002

상태: 초안 작성
구분: 여행기 기록
기능 간단 설명: 여행 기록 중, 사용자 주변 관광지의 GPS 좌표를 기반으로 이동 경로를 기록
기능명: 여행 이동 위치 기록
우선 순위: 1순위

| 항목 | 내용 |
| --- | --- |
| **기능 ID** | RECD_002 |
| **기능 명** | 여행 이동 위치 기록 |
| **기능 설명** | 사용자의 위치 변화를 실시간으로 감지하여, 일정 거리 이상 이동하거나 일정 시간 이상 체류 시 자동으로 여행 기록을 생성한다. 
이동 및 체류의 조건은 사전에 정의된 기준에 따라 판단되며, 여행 타임라인 구성의 기준이 된다. 
단, 위치 데이터는 사용자의 실제 GPS 좌표 대신, 반경 내 관광지 데이터를 기준으로 근사 처리하여 개인정보 보호를 준수한다. |
| **사용자 액션** | 여행을 기록 중인 상태일 때 자동 동작 |
| **시스템 동작** | 1. 백그라운드 위치 추적 활성화
2. GPS 좌표를 일정 주기(예: 1분)에 따라 수신
3. 마지막 위치와 현재 위치의 거리 비교
4-1. 이동: 일정 거리(예: 200m) 이상 이동하면 ‘이동 기록’ 생성
4-2. 체류: 동일 위치에서 일정 시간 이상(예: 10분) 체류 시 ‘체류 기록’ 생성
5. 개인정보 보호를 위해, 수신된 좌표 자체는 저장하지 않으며, 반경 300m 내 가장 가까운 관광지를 TourAPI로 조회하여 해당 장소 정보만 기록 (예: 속초해변, 거리: 220m, 체류: 13분) 
6. 생성된 이동/체류 기록은 여행기 ID에 연동되어 타임라인에 저장됨

이동 기록의 기준(이동 거리 or 체류) 이야기 해봐야 할 듯
→ 체류로 알고 있긴 함 - 체류의 기준 정하기(시간, 근사 거리) 
→ (체류 선택 시) 근처 카페갔는데 거리 근사 처리되어서 관광지에 대한 처리를 한다면 사용자 만족도가 감소할 듯 ,, ? 실제 그 관광지를 가지는 않음. 하지만 타임라인에는 해당 관광지를 방문한 위치 정보와 감상평(카페)이 남을 것임  
  |
| **입력 데이터** | 위치 좌표(GPS), 시간 |
| **출력 데이터** | `travel_log_entry` 객체
→ `travel_id`, `place_name`, `place_id`, `matched_distance`, `stay_time`, `entry_type`(`이동`/`체류`) 등
→ GPS 좌표 저장 x  |
| **상태 전이** | 없음 (단, 기록 상태가 `recording`일 때만 작동) |
| **예외 처리** | 위치 수신 불가(GPS OFF): 위치 권한 요청 or 에러 메시지
사용자가 이동하지 않음: 기록 없음
백그라운드 제한 상태: 사용자에게 권한 재설정 유도 |
| **비고** | 백그라운드 추적은 OS 제약 사항(Android/iOS)에 따라 다를 수 있어 구현 전 기술 조사 필요. 예외 상황에서도 UX 손실이 없도록 안내 필요.
TourAPI를 활용한 위치 근사화 처리 로직 필요(최소 거리 기준 매칭)

**관광지 정보가 없는 지역에 대한 UX 설계 필요 (예: "이름 없는 장소")
→ 매칭할 장소가 없으면, 기록할 ‘장소 이름’도 없는 상황 발생 (TourAPI 없는쪽으로 가면 위치 어떻게 저장할겨 ,<? )** |

## ☑️ 개발자/기획자 참고사항

---

| 역할 | 주의 사항 |
| --- | --- |
| **백엔드** | 이동/체류 기록은 travel_log 테이블 또는 위치 로그 테이블에 연동되어야 하며, travel_id를 키로 묶어야 함. 
위치별 타임스탬프 기록 필수. |
| **프론트엔드** | OS별 위치 권한 상태를 확인하고, 사용자가 위치 권한을 철회하거나 배터리 제한을 설정할 경우 UX 메시지 제공 필요. |
| **디자이너** | 실시간으로 자동 기록된 이동/체류 로그가 타임라인에 자연스럽게 녹아들도록 UI 설계 필요. |
| **AI 처리** | 위치 기반 데이터를 분석하여, 감정 문장/태그와 연계할 때 방문 장소 정보와 날씨 데이터 등도 병합 처리 가능 (예: “해변에서 10분 이상 머문 기록” → 감정 추론) |

### 📌 시퀀스 다이어그램 (요약 흐름)

```

[위치 센서] → [앱] → 위치 좌표 수신
→ 거리/체류 조건 계산 → 조건 만족 시 기록 생성
→ travel_log에 저장 → 타임라인 UI 업데이트
```

### 📍 위치 추상화 알고리즘

```
1. 사용자의 실시간 GPS 좌표 수신
2. 반경 N 미터 이내 관광지 리스트 조회 (TourAPI 장소 검색 API)
3. 가장 가까운 관광지를 선택
4. 그 관광지를 기준으로 ‘사용자 체류 기록’ 저장
```

> 예시: latitude=38.205, longitude=128.591 → 반경 500m 내 검색 → 가장 가까운 장소가 "설악 해맞이공원"일 경우 해당 이름만 기록
>
